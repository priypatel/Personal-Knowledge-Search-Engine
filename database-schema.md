# Database Schema

**Project:** Knowbase — Personal Knowledge Search Engine
**Version:** 1.0 MVP
**Date:** March 2026
**Database:** PostgreSQL (relational + pgvector for embeddings)

---

## 1. PostgreSQL — Unified Schema (Relational + Vector)

All data — structured metadata and vector embeddings — lives in a **single PostgreSQL database** using the **pgvector** extension for similarity search.

### 1.0 Enable pgvector Extension

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

### 1.1 Entity Relationship Overview

```
users
  │
  ├─── documents (one-to-many: user has many documents)
  │       │
  │       ├─── suggestions (one-to-many: document has many suggestions)
  │       │
  │       └─── document_embeddings (one-to-many: document has many chunks/vectors)
  │
  └─── chat_sessions (one-to-many: user has many chat sessions)
          │
          └─── chat_messages (one-to-many: session has many messages)
                    │
                    └─── message_citations (one-to-many: message references many chunks)
```

### 1.2 Table: `users`

Stores registered users. Admin accounts are seeded via migration — not UI-created.

```sql
CREATE TABLE users (
  id          UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  email       VARCHAR(255)  NOT NULL UNIQUE,
  password    VARCHAR(255)  NOT NULL,               -- bcrypt hash
  name        VARCHAR(100),
  role        VARCHAR(20)   NOT NULL DEFAULT 'user', -- 'user' | 'admin'
  created_at  TIMESTAMP     NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMP     NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_users_email ON users(email);
```

**Field Notes:**
- `id`: UUID primary key (prevents enumeration attacks)
- `password`: bcrypt-hashed, never stored in plain text
- `role`: `'user'` for normal users, `'admin'` for seeded admin accounts

### 1.3 Table: `documents`

Tracks every uploaded document and its processing state.

```sql
CREATE TABLE documents (
  id            UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID          NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  filename      VARCHAR(255)  NOT NULL,
  original_name VARCHAR(255)  NOT NULL,
  file_type     VARCHAR(10)   NOT NULL CHECK (file_type IN ('pdf', 'docx', 'txt')),
  file_size     INTEGER       NOT NULL,              -- bytes
  file_path     VARCHAR(512)  NOT NULL,              -- disk path or object storage key
  status        VARCHAR(20)   NOT NULL DEFAULT 'processing', -- 'processing' | 'ready' | 'error'
  chunk_count   INTEGER,                             -- populated after processing
  summary       TEXT,                                -- generated by LLM after processing
  error_msg     TEXT,                                -- populated if status = 'error'
  created_at    TIMESTAMP     NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMP     NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_documents_user_id ON documents(user_id);
CREATE INDEX idx_documents_status  ON documents(status);
```

**Status Flow:** `processing` → `ready` | `error`

### 1.4 Table: `document_embeddings` (pgvector)

Stores text chunk embeddings for semantic search. This replaces the need for a separate vector database.

```sql
CREATE TABLE document_embeddings (
  id            UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  document_id   UUID          NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  user_id       UUID          NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  chunk_index   INTEGER       NOT NULL,
  chunk_text    TEXT          NOT NULL,
  embedding     vector(1536)  NOT NULL,              -- pgvector column
  filename      VARCHAR(255)  NOT NULL,
  file_type     VARCHAR(10),
  page_ref      VARCHAR(50),                         -- page number or section reference
  char_start    INTEGER,
  char_end      INTEGER,
  created_at    TIMESTAMP     NOT NULL DEFAULT NOW()
);

-- HNSW index for fast approximate nearest neighbor search (cosine distance)
CREATE INDEX idx_embeddings_hnsw ON document_embeddings
  USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);

-- B-tree indexes for filtering
CREATE INDEX idx_embeddings_user_id     ON document_embeddings(user_id);
CREATE INDEX idx_embeddings_document_id ON document_embeddings(document_id);
```

**pgvector Search Query:**
```sql
-- Find top-5 most similar chunks for a user's query
SELECT
  id, document_id, chunk_index, chunk_text, filename, page_ref,
  1 - (embedding <=> $1::vector) AS similarity_score
FROM document_embeddings
WHERE user_id = $2
  AND 1 - (embedding <=> $1::vector) >= 0.70  -- score threshold
ORDER BY embedding <=> $1::vector
LIMIT 5;
```

**Notes:**
- `embedding <=>` is the pgvector cosine distance operator
- `1 - distance = similarity` (cosine similarity from cosine distance)
- HNSW index provides fast ANN search; `m=16, ef_construction=64` is a good default for small-to-medium datasets
- For MVP scale (< 500K vectors), HNSW performs well without partitioning

### 1.5 Table: `suggestions`

Stores AI-generated suggested questions, cached per document.

```sql
CREATE TABLE suggestions (
  id           UUID       PRIMARY KEY DEFAULT gen_random_uuid(),
  document_id  UUID       NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  user_id      UUID       NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  question     TEXT       NOT NULL,
  sort_order   SMALLINT   NOT NULL DEFAULT 0,    -- display order (0, 1, 2)
  created_at   TIMESTAMP  NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_suggestions_document_id ON suggestions(document_id);
CREATE INDEX idx_suggestions_user_id     ON suggestions(user_id);
```

**Notes:**
- 3 suggestions generated per document
- Cached: once generated, not regenerated unless document is re-processed
- Redis also caches active suggestions for the authenticated user

### 1.6 Table: `chat_sessions`

Each chat session is a named conversation for a user.

```sql
CREATE TABLE chat_sessions (
  id          UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title       VARCHAR(200),                         -- auto-generated from first query
  created_at  TIMESTAMP    NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMP    NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_chat_sessions_user_id    ON chat_sessions(user_id);
CREATE INDEX idx_chat_sessions_updated_at ON chat_sessions(updated_at DESC);
```

---

### 1.7 Table: `chat_messages`

Stores individual messages within a chat session (user and AI turns).

```sql
CREATE TABLE chat_messages (
  id          UUID       PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id  UUID       NOT NULL REFERENCES chat_sessions(id) ON DELETE CASCADE,
  user_id     UUID       NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role        VARCHAR(10) NOT NULL CHECK (role IN ('user', 'assistant')),
  content     TEXT       NOT NULL,
  created_at  TIMESTAMP  NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_chat_messages_session_id ON chat_messages(session_id);
CREATE INDEX idx_chat_messages_created_at ON chat_messages(created_at);
```

---

### 1.8 Table: `message_citations`

Links each AI message to the embedding chunks used to generate it.

```sql
CREATE TABLE message_citations (
  id              UUID       PRIMARY KEY DEFAULT gen_random_uuid(),
  message_id      UUID       NOT NULL REFERENCES chat_messages(id) ON DELETE CASCADE,
  document_id     UUID       NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  embedding_id    UUID       REFERENCES document_embeddings(id) ON DELETE SET NULL,
  chunk_index     INTEGER    NOT NULL,
  chunk_text      TEXT,                     -- snapshot of the chunk text
  page_ref        VARCHAR(50),
  relevance_score FLOAT,                    -- cosine similarity score
  created_at      TIMESTAMP  NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_message_citations_message_id  ON message_citations(message_id);
CREATE INDEX idx_message_citations_document_id ON message_citations(document_id);
```

## 2. Redis — Cache & Queue Schema

### 2.1 BullMQ Job Queues

| Queue Name | Job Data | Retry | TTL |
|---|---|---|---|
| `document-processing` | `{ documentId, filePath, userId }` | 3× exponential backoff | N/A (persistent) |

### 2.2 Cache Keys

| Key Pattern | Value | TTL |
|---|---|---|
| `suggestions:{userId}` | JSON array of suggestion objects | 24 hours |
| `doc-status:{documentId}` | `'processing'` \| `'ready'` \| `'error'` | 1 hour |

## 3. Migration Strategy

### 3.1 Schema Migrations (Knex.js)

- All schema changes managed via Knex migration files in `packages/api/src/db/migrations/`
- First migration must enable the pgvector extension: `CREATE EXTENSION IF NOT EXISTS vector`
- Run on deployment: `npx knex migrate:latest`

### 3.2 Seed Files

- Admin user creation via seed: `packages/api/src/db/seeds/01_admin_user.js`
- Admin accounts are **never** created through the UI
- Seed command: `npx knex seed:run`

## 4. Data Retention & Privacy

| Data Type | Policy |
|---|---|
| User account | Retained until user deletion request |
| Documents | Retained; user can delete (cascades to embeddings, suggestions, citations) |
| Document embeddings | Deleted when parent document is deleted (ON DELETE CASCADE) |
| Chat sessions | Retained; user can delete individual sessions |
| Redis cache | Auto-expires per TTL |
| Uploaded files | Stored at `uploads/{userId}/{documentId}/{filename}` |

## 5. pgvector Performance Notes

| Consideration | Value / Recommendation |
|---|---|
| Index type | **HNSW** (preferred for recall quality) or **IVFFlat** (alternative, faster build) |
| HNSW params | `m=16, ef_construction=64` for default; increase `ef_construction` for higher recall |
| Distance operator | `<=>` for cosine distance; `<->` for L2; `<#>` for inner product |
| Max vectors (MVP) | ~500K vectors — HNSW handles well within a single PostgreSQL instance |
| Scaling beyond MVP | Consider partitioning `document_embeddings` by `user_id` for large multi-user scale |
| PostgreSQL version | Requires PostgreSQL 15+ with pgvector 0.5+ extension installed |
